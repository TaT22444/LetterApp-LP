# Chocoleta!? ローンチ戦略

## 概要

ウェイトリスト（事前登録）→ 段階的リリース → バイラルループ（手紙送信 = 招待）→ 一般公開、の4段階でユーザーを拡大する。

---

## フェーズ構成

### フェーズ1: プレローンチ（現在）

**目的**: リリース前にユーザーリストを蓄積する

- LP（chocoleta.netlify.app）で事前登録を受け付ける
- メールアドレスを Firebase Auth + Firestore に保存
- 確認メール（メールリンク認証）で本人確認済みのリストを構築
- Firestore の `createdAt`（serverTimestamp）で登録順を記録済み

**現状の実装で完了していること**:
- メール入力 → 確認メール送信 → リンククリックで登録完了
- 登録済みユーザーの判定（Cloud Function `checkPreRegistration`）
- プライバシーポリシー・利用規約
- 重複登録防止（uid ベースの setDoc）

**LP の表記**: 「事前登録」のまま。日本のユーザーに馴染みがあり、内部的にはウェイトリスト（登録順管理）として運用可能。

---

### フェーズ2: 段階的リリース

**目的**: サーバー負荷を制御しつつ、初期ユーザーの体験品質を確保する

**開放の単位**:
- Firestore の `createdAt` 順に、バッチ単位（例: 100人ずつ）で開放
- 各バッチに「利用開始案内メール」を送信

**開放メールの内容**:
- アプリのダウンロードリンク（App Store / Google Play）
- 初回ログイン方法の案内
- 「あなたは○番目の登録者です」のような特別感の演出（任意）

**判断基準（次のバッチを開放するタイミング）**:
- サーバー負荷が安定しているか
- 致命的なバグ報告がないか
- ユーザーからのフィードバックを反映できたか

**想定スケジュール（目安）**:
- 最初の100人: リリース直後
- 次の100人: 1〜2日後（安定確認後）
- 以降: 状況に応じて加速（500人、1000人単位へ）

---

### フェーズ3: バイラル拡大

**目的**: ユーザーの自然な利用行動によって新規ユーザーを獲得する

**仕組み**:

```
利用者A（ウェイトリスト経由でアプリ利用中）
  ↓ アプリで手紙を作成
  ↓ LINEやSNSで手紙リンクを送信
  ↓
受け取り側B（アプリ未インストール）
  ↓ ブラウザで手紙を開封（3Dアニメーション体験）
  ↓ 「自分も送りたい」→ アプリDLページへの導線
  ↓ アプリインストール＆アカウント作成
  ↓ 自動的に利用開始（ウェイトリスト不要）
```

**重要な設計原則**:

1. **受け取り側はアプリ不要**: 手紙の開封はWebリンクでブラウザ上で完結する。これがバイラルの障壁を最小化する最重要ポイント。

2. **手紙を送る行為 = 招待**: 別途「招待枠」のような仕組みは設けない。手紙を送ること自体がサービスの紹介になるため、自然な形で広がる。

3. **招待枠やシェア特典は設けない**: 「SNSでシェアしたら順番が上がる」等の仕組みはスパム感が出て、Chocoleta の温かいブランドイメージに合わないため採用しない。

**バイラル係数の期待値**:
- 1人のユーザーが平均2〜3人に手紙を送ると仮定
- そのうち10〜20%がアプリをDLすると仮定
- バイラル係数 = 0.2〜0.6（自然成長が期待できる水準）

---

### フェーズ4: 一般公開

**目的**: ウェイトリストの全員を開放し、誰でも利用可能にする

**タイミング**:
- ウェイトリスト登録者全員への開放が完了した時点
- サーバー・インフラが十分にスケール可能と確認できた時点
- 致命的なバグが解消されている時点

**対応事項**:
- App Store / Google Play での一般公開
- LP の「事前登録」を「ダウンロード」ボタンに切り替え（コメントアウトしてある Store ボタンを復活）
- ウェイトリスト関連の Cloud Function・Firestore ルールのクリーンアップ

---

## 技術的な補足

### 現在のデータ構造（変更不要）

Firestore `pre-registrations/{uid}`:
```json
{
  "email": "user@example.com",
  "verified": true,
  "createdAt": "2026-02-23T...",
  "verifiedAt": "2026-02-23T...",
  "uid": "firebase-auth-uid"
}
```

- `createdAt` で登録順のソートが可能
- `uid` で Firebase Auth ユーザーと紐付け済み
- フェーズ2で「開放済みか」を管理するフィールド（例: `invited: true`, `invitedAt: timestamp`）を追加すれば良い

### フェーズ2で追加が必要になるもの

- 開放メール送信用の Cloud Function またはスクリプト
- Firestore に `invited` フィールドを追加するマイグレーション
- アプリ側の認証フロー（事前登録メールでのログイン連携）

### フェーズ3で追加が必要になるもの

- 手紙の開封ページ（Web）→ アプリDLへの導線
- アプリ内でのアカウント作成フロー
- 手紙リンクからアプリへのディープリンク対応

---

## やらないこと

| やらないこと | 理由 |
|---|---|
| 招待枠（1人5枚など） | 管理が複雑。手紙送信自体が招待と同義 |
| SNSシェアで順番繰り上げ | スパム感がブランドに合わない |
| 有料の優先枠 | プレローンチ段階での課金はユーザー体験を損なう |
| ウェイトリスト順位の公開表示 | 開発コストに対してリターンが薄い。やるなら「○番目に登録」程度 |
| LP 表記の「ウェイトリスト」への変更 | 日本語ユーザーには「事前登録」のほうが自然 |
